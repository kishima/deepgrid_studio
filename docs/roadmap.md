# DeepGrid Studio 実装ロードマップ

全体仕様は [project.md](project.md) を参照。各フェーズの詳細は planN.md として
起こし、実装者は planN.md 単体で着手できる粒度まで落とす。
フェーズ順は「遊べるランタイムを先に育て、その上にエディターを載せる」方針。
各 plan の完了条件には必ず docker での自動検証(DEEPGRID_DEBUG_SHOT)を含める。

## 全体像

DeepGrid Studio は2つの顔を持つ:

1. **ランタイム(プレイヤー)**: 作られたゲームを遊ぶ実行系
2. **スタジオ(エディター)**: ゲームを作る編集系(マップ/キャラ/アイテム/
   モンスター/魔法/イベント/デモの各エディター)

両者は同じデータモデル(RONベースのプロジェクト形式)を共有する。
数量上限はすべて `LimitsConfig` 経由(project.md「上限値の扱い」)。

## 依存関係と順序の理由

- **plan2(ダンジョン構造)を最優先**: 高さ・落下はこのゲームの核となる差別化
  要素であり、データモデルの形(フロア積層)が他の全plan に影響する。
  エディターより先に確定させる。
- **plan3(プロジェクト形式+マップエディター)を早期に**: 以降のplanのテスト
  データを手書きRONではなくエディターで作れるようになり、開発が加速する。
  Undo/Redo 等の編集基盤もここで確立する(project.md「UIの方針」)。
- **plan4→7 はランタイムの積み上げ**: キャラ(HP等の器)→アイテム(装備が
  戦闘の前提)→モンスター(戦闘相手)→魔法(戦闘の拡張)の順で、
  常に「遊べる状態」を保ったまま足す。
- **plan8(イベント)は戦闘後**: イベントの多く(モンスター発生、パーティ復活
  等)がキャラ・モンスターの存在を前提とするため。
- **plan9(エディター群)はデータモデル安定後**: ランタイム側の
  パラメーターが出揃ってからUIを作ることで手戻りを防ぐ。

## 横断ルール(全planで共通)

- 実装指示書(planN.md)には スコープ/やらないこと/成果物/実装ステップ/
  受け入れ基準/検証手順 を必ず含める(plan1.md の構成を踏襲)。
- 数量上限のハードコード禁止。`LimitsConfig` を唯一の情報源とする。
- 検証は docker 経路が基準。`DEEPGRID_DEBUG_SHOT` を新機能が写る形で拡張し、
  受け入れ基準に画像確認を含める。生成物は mtime を確認する。
- ホストでの `cargo clippy`(`CARGO_TARGET_DIR` 指定)が警告なしで通ること。
- Bevy 0.15 固定。バージョン更新は独立した plan として行う(機能planに混ぜない)。
