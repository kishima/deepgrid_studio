# plan2: ダンジョン構造の完成 — 複数フロア・落下・はしご・ドア

## この文書について

DeepGrid Studio の第2実装計画書。前提知識は [project.md](project.md)(仕様)、
[plan1.md](plan1.md)(基盤。**実装済み**)、[roadmap.md](roadmap.md)(全体計画)。
plan1 と同様、実装者はこの文書だけで着手できることを意図している。
開発環境の制約(docker 必須、CARGO_TARGET_DIR 等)は plan1.md
「開発環境の前提」をそのまま引き継ぐ。必ず先に読むこと。

## ゴール

だんだんダンジョンの核である**立体ダンジョン**を動かす:
1レベル=複数フロア(既定5)の積層構造の中を、はしごで昇降し、
床のない場所では**落下**する。ドアの開閉と一方通行ブロックも入れ、
「ダンジョンの地形要素」をこの plan で完成させる。

## 高さ・支持のルール(DeepGrid Studio としての定義)

オリジナルの内部仕様は正確には不明なため、以下を本プロジェクトの定義とする
(実装中に不自然さが見つかったら、実装を変える前にこの節を更新して合意を取る):

- 1レベルはフロア0(最下層)からフロア N-1(最上層)の積層。
  各フロアは独立した width×height のブロックグリッド。
- **足場の判定**: フロア f のセル (x,y) に立てるのは、
  「f == 0(最下層は岩盤の上)」または「フロア f-1 の同セルが `Wall`(壁の上面が
  床になる)」のとき。
- **落下**: 立てないセルに進入したら、立てるフロアまで即座に落下を開始する。
  複数フロアぶん連続で落ちることもある。落下は project.md「UIの方針」に従い
  滑らかなアニメーション(1フロアあたり 0.15秒程度、加速感はイージングで表現)。
  落下ダメージは plan4(HPの導入)まで実装しない。
- **はしご**: `Ladder` ブロック。はしごセルでは立ち位置の支持がなくても
  落下しない。同セルの直上フロアもはしごなら上へ、直下フロアがはしご
  (または進入可能)なら下へ移動できる。
- **天井**: フロア f の頭上は、フロア f+1 の同セルが `Wall` なら塞がっている。
  最上層の頭上は常に岩盤。はしごで上フロアへ移動できるのは頭上が
  塞がっていない(= 上フロアの同セルが進入可能な)ときのみ。

### 追補: オリジナル詳細仕様との差分(2026-07-13)

[dandan_spec_mapeditor.md](dandan_spec_mapeditor.md) で判明した差分。
plan2 実装は変更せず、以下を後続 plan(主に plan8)で取り込む:

- **穴ブロック**: オリジナルには「下のフロアへ落下する空間」が明示的な
  ブロックとして存在する(2Dエディットのみ設定可)。本プロジェクトの
  「支持がなければ落下」ルールに加えて、**支持があっても床が無い
  `Block::Hole`** を追加する(描画は床タイルを張らない、進入すると落下)。
- **ホロスコープは全12種類**: 水平4方向だけでなく上・下を含む12方向の
  通り抜け制御がある。plan2 実装は水平4方向のみ。垂直方向の通行制御は
  上下移動ギミックと合わせて拡張する。
- **ドアは配置時に開閉状態を指定**(CLOSE/OPEN のパーツが別)。plan2 は
  「kind ごとに一括開閉、初期状態は閉」なので、マップ形式にドアの
  初期状態を持たせる拡張が必要。
- **連絡通路**: レベル(階)間を移動する専用の通路概念がある。
  レベル間移動の仕組みとして plan8 で扱う。

## スコープ

### やること

1. データモデルとマップ形式の複数フロア対応
2. 描画の複数フロア対応(吹き抜け・穴が見える)
3. 落下とはしご昇降(滑らかアニメーション付き)
4. ドア(2種、開閉状態、正面のドアを開閉する操作)
5. ホロスコープブロック(特定方向からのみ通過可能な一方通行ブロック)
6. 液体ブロック(水・火・毒)の見た目(半透明の色付き表現)。
   ダメージ判定は plan5(耐性パラメーター導入後)
7. 検証スクリーンショットの複数シーン対応

- 落下・液体のダメージ(HP がまだ無い) → plan4/plan5
- 鍵穴・スイッチ・しかけ床・ワープ・書き込める壁 → plan8
  (ドアの「状態」だけ先に作り、開閉トリガーの拡張は plan8 で行う)
- 複数レベル(レベル間移動) → plan8(ワープ/イベントと同時)
- モンスター・アイテム配置 → plan5/plan6
- サイクル時間システム → plan4

## データモデルの変更

`src/dungeon/block.rs`:

```rust
pub enum Block {
    Wall,
    Empty,
    Water,
    Fire,
    Poison,
    Ladder,
    /// ドア。kind は 0 or 1(1レベルあたりの種類数は LimitsConfig 経由)。
    Door { kind: u8 },
    /// 一方通行。pass_from の方向からのみ進入・通過できる。
    Horoscope { pass_from: Facing },
}
```

- ドアの開閉「状態」はブロック(地形)ではなく `DoorStates` リソース
  (レベル内のドアkindごとの開/閉。オリジナル準拠で「ドア1」「ドア2」が
  それぞれ一括開閉)として持つ。地形データは不変、状態は可変、と分離する。
  ※オリジナルは個別ドアではなく種類単位の開閉なのでこの構造でよい。
- `GridPos.floor` は plan1 で導入済み。移動・描画・判定のすべてで
  実際に使うようにする。

`assets/maps/` のマップ形式 v2:

```ron
(
    width: 40, height: 40,
    start_x: 3, start_y: 3, start_floor: 2, start_facing: East,
    // floors[0] が最下層。各フロアは rows(height 本の文字列)。
    floors: [
        (rows: [ ... ]),
        (rows: [ ... ]),
    ],
)
```

- 文字の追加: `H`=Ladder, `1`/`2`=Door(kind 0/1),
  `<` `>` `n` `v`=Horoscope(それぞれ West/East/North/South から進入可)。
- 旧形式(単一 rows)は捨ててよい。test_level.ron を v2 に書き換える。

## 実装ステップ

### Step 1: マップ形式 v2 とローダー

- `loader.rs` を v2 対応に書き換え、全フロアを `Level.floors` に読み込む。
- バリデーション追加: フロア数が `LimitsConfig.floors_per_level` 以下、
  start_floor が範囲内、start 位置が上記「足場の判定」を満たすこと。
- テストマップを3フロア構成で作り直す。必ず含める要素:
  吹き抜け(上のフロアの穴から下の部屋が見える)、2フロア分の連続落下が
  起きる穴、上下に貫通するはしご、ドア1/ドア2 が各1枚以上、
  ホロスコープブロックの一方通行通路、水たまり。

### Step 2: 描画の複数フロア対応

`render/dungeon_mesh.rs` を作り直す:

- フロア f のジオメトリを y オフセット `f * BLOCK_SIZE` に配置する。
- plan1 の「全面床スラブ+全面天井スラブ」を廃止し、**セル単位の床タイル**に
  変える: フロア f のセルの床面は「支持あり(下が Wall または f==0)」の
  ときだけ張る。天井面は「最上層の頭上」のみ全面に張る。
  これで穴・吹き抜けが自然に表現される。
- ドア: 開=枠だけ(通れる見た目)、閉=壁より明るい色の薄い板。
  `DoorStates` の変更でメッシュの表示/非表示を切り替える
  (エンティティに kind を持たせ、Visibility を切り替えるのが簡単)。
- はしご: 細い縦板や柱の簡易表現でよい(テクスチャは plan10)。
- 液体: 床上に半透明(alpha 0.5 程度)の色付き薄板
  (水=青、火=橙、毒=緑)。`AlphaMode::Blend` を使う。
- ホロスコープ: 通過可能方向が分かる見た目(通過可の面だけ色を変えた
  薄板など)。凝らなくてよいが、方向が画面から判別できること。

### Step 3: 移動判定の拡張

`player/movement.rs` の `start_action` 周辺:

- 進入可否をブロック種別で判定する関数を `Block` に集約する:
  `Wall`=不可、`Door`=`DoorStates` が開なら可、`Horoscope`=一方通行、
  その他=可。
  - **ホロスコープの一方通行の定義(実装に合わせて更新。2026-07-12)**:
    `pass_from` は「通過を許す進行方向」とする。ホロスコープセルへは
    `pass_from` 方向に進むときのみ**進入**でき、セルから出るのも
    `pass_from` 方向のみ。これで `pass_from` 方向にだけ通り抜けられる
    「弁」になる(マップ字形 `< > n v` の矢印が示す向き = 通れる向き)。
    ※当初の指示文にあった「出る方向は pass_from の逆方向のみ」は、
      進入と組み合わせると入った側へ戻るだけの袋小路になり一方通行に
      ならないため、上記(進入・退出とも pass_from 方向)に改めた。
    実装は `Block::allows_exit` / `Block::allows_enter`(いずれも
    進行方向 `dir` を取り、`dir == pass_from` を要求)。
- 進入成功後に「足場の判定」を行い、支持がなければ落下処理へ。

### Step 4: 落下とはしご

- 落下: 進入アニメーション完了後、支持のあるフロアまで 1フロアずつ
  落下アニメーション(0.15秒/フロア、ease-in で加速感)を連結する。
  落下中は入力を受け付けない(バッファもクリア)。
- はしご昇降: キーは `R`=上る / `F`=下りる。
  上る条件・下りる条件は「高さ・支持のルール」の節のとおり。
  昇降も 0.25秒の滑らか補間(垂直移動)。
- カメラの y は常に `floor * BLOCK_SIZE + EYE_HEIGHT`。

### Step 5: ドアの開閉操作

- キー `Space`: 正面1マスがドアなら、その **kind の開閉状態をトグル**する
  (オリジナル準拠: ドア1/ドア2 は種類単位で一斉に開閉)。
- 閉じたドアのセルにいる状態で閉じられても plan2 では考慮しない
  (ドアセル進入は開状態のみ可なので、閉→開→進入→閉で挟まれるケースは
  プレイヤー自身の操作でしか起きない。挟まれ死は HP 導入後に扱う)。

### Step 6: 検証スクリーンショットの複数シーン化

- `DEEPGRID_DEBUG_SHOT` を値で分岐させる:
  - `1` : 従来どおり開始位置
  - `fall` : 落下穴の手前まで自動で歩き、落下後の下フロアで撮影
  - `ladder` : はしごを1フロア上って撮影
  - `door` : ドアの正面で開閉トグル後(開いた状態)に撮影
- 実装は「起動時にアクション列(`Vec<Action>` + 昇降/開閉)を積んで
  順に自動実行し、完了後に撮影して終了」とする。手動操作の代替であり、
  移動ロジック自体を経由すること(専用のテレポートで済ませない)。

## 受け入れ基準

1. `./docker/deepgrid-build.sh` 完走、ホスト `cargo clippy` 警告なし。
2. `DEEPGRID_DEBUG_SHOT=1` : 3フロア版マップの開始地点が写る。
   吹き抜け越しに下のフロアが見えるアングルであること。
3. `DEEPGRID_DEBUG_SHOT=fall` : 落下後の下フロアの画像が撮れ、
   カメラ高さが下フロア相当になっている。
4. `DEEPGRID_DEBUG_SHOT=ladder` : 上フロアからの画像が撮れる。
5. `DEEPGRID_DEBUG_SHOT=door` : 開いたドアが写る。
6. 手動確認(人間が実施): 落下・昇降・ドア開閉・一方通行が
   滑らかなアニメーションで動作し、壁抜け・意図しない落下がない。
7. すべてのスクリーンショットで mtime が当該実行時刻であること。

## 実装上の注意

- フロア数・ドア種類数を定数にしない(`LimitsConfig` 経由)。
- 「高さ・支持のルール」と実装が食い違ったら、実装に合わせて仕様を
  黙って変えない。この文書を更新して差分を明記すること。
- 描画は plan1 同様の割り切り(単色マテリアル、面カリングなし)を維持する。
  ただし床をセル単位タイルにするため描画物は増える。40×40×5 で顕著に
  重い場合のみ、同一フロアの連続床タイルの矩形マージ程度までは行ってよい
  (それ以上の最適化はしない)。
- 半透明(液体)は不透明ジオメトリの後に描画されるが、複数の半透明が
  重なるアングルで乱れが見えても plan2 では許容する。
- カメラの `ClusterConfig::Single` は維持する(plan1 でのライト矩形
  アーティファクト対策。movement.rs のコメント参照)。
