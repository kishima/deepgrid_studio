# DeepGrid Studio プロジェクトについて

DeepGrid Studioは、かつてPC98でツクールシリーズの一つとして発売された「だんだんダンジョン」をオマージュしたプロジェクトである。

## だんだんダンジョン概略仕様

『だんだんダンジョン』は、リアルタイムで進行する本格的な3DダンジョンRPGを作成できるソフトです。作成可能なゲームの仕様を項目別に列挙します。

より詳細な仕様は以下の別紙にまとめている。本書と食い違う場合は別紙(詳細)を優先する:
[dandan_spec_event.md](dandan_spec_event.md)(イベント設定)/
[dandan_spec_mapeditor.md](dandan_spec_mapeditor.md)(マップエディター)/
[dandan_spec_things_editor.md](dandan_spec_things_editor.md)(キャラ・アイテム・魔法エディター)

見た目や大まかな概念は「ダンジョンマスター」に近いです。大きな違いは、内部で高さの概念があり、落下などが発生することです。

### 1. ダンジョン構造の仕様
*   **レベルとフロア**: 全14の「レベル」で構成され、1つのレベルは5つの「フロア」が重なった立体的な構造をしています。
*   **フロアサイズ**: 1つのフロアは、**40×40ブロック**の広さを持ちます。
*   **構成パーツ**:
    *   **基本属性**: 通常の壁、通路（空）、水、火（溶岩）、毒の5つの基本ブロックがあります。液体ブロック（水・火・毒）に入ると、キャラの耐性パラメーターが低い場合に1サイクルにつき1ダメージを受けます。
    *   **ドア**: 1レベルにつき最大2種類（「ドア1」「ドア2」）設定でき、それぞれに開・閉の状態があります。
    *   **特殊ブロック**: ホロスコープブロック（特定の方向からのみ通り抜け可能）、はしご（上下フロア間の移動）、書き込める壁（メッセージを表示可能）があります。
    *   **ギミック**: 鍵穴、スイッチ、しかけ床（踏むと発動）、ワープポイントが設置可能です。

### 2. キャラクターの仕様
*   **作成数**: 最大20人までキャラクターを登録・作成できます。
*   **パーティ構成**: 作成した20人の中から最大4人のパーティを編成してプレイします。
*   **パラメーター**: 名前、性別、年齢、経歴などのプロフィールのほか、以下の能力値を設定できます。
    *   **基本能力**: レベル（最大255）、MHP、MMP、攻撃力、防御力、すばやさ、遠投力、運搬力、肺活量、耐熱力、耐毒性、魔法知識、集中力、鑑定力、盗みのうで、歯の強さなど。
    *   **成長タイプ**: 5種類（平均型、早期開花型、大器晩成型、天才型、才能なし）から選択でき、成長の仕方が異なります。

### 3. アイテムの仕様
*   **作成・配置数**: 最大199種類を作成可能で、1つのレベルにつき合計1023個まで配置できます。
*   **パラメーター**: するどさ（武器の威力）、かたさ（防具の性能）、おもさ、温度、栄養価、エントロピー、対魔法力、対鑑定力、投げやすさ、持ちやすさ、収納数などが細かく設定可能です。
*   **アイテム種別**: 一般アイテム、巻物、空容器、液体、照明、マップ、コンパス、潜望鏡、鉛筆、装飾品など14種類が用意されています。
*   **変形条件**: 温度の変化などによって、特定のアイテムに自動変化する設定（12パターン）が可能です。

### 4. モンスターの仕様
*   **作成・配置数**: 最大56種類を作成できます。1つのレベルに登場させられるのはそのうち4種類までで、配置合計数は**最大48体**という制限があります。
*   **パラメーター**: 最大HP、攻撃力、防御力、すばやさ、攻撃頻度、視界（認識距離）、逃避HP（逃げ始める体力）などが設定可能です。
*   **移動とサイズ**: 移動タイプ（地上・空中・なし）や通路幅の認識を設定できます。サイズは通常サイズ（1×1）と倍サイズ（2×2）の2種類があります。

### 5. 魔法の仕様
*   **作成数**: 最大80種類を作成できます。
*   **パラメーター**: 使用MP、難易度（習得に必要な魔法知識レベル）、効果時間、光弾数（魔法のグラフィックが飛ぶ数）、液体化の可否、効果音などが設定可能です。
*   **魔法種別**: 能力値の増減、MPの変更、キャラクターの復活、ダンジョン内を明るくする照明魔法など27種類から選べます。

### 6. イベント・システム仕様
*   **イベントの種類**: 全14種類。ワープ、水位変更、BGM変更、パーティ復活、モンスター/アイテムの発生、ドアの開閉、デモ起動、スイッチ操作、フラグ変更、ループ設定などが可能です。
*   **発動条件とフラグ**: 0〜63サイクルの実行遅延を設定できるほか、**64個のイベントフラグ**を利用し、AND/OR条件による複雑な発動管理が行えます。
*   **デモ作成**: オープニング、エンディング（各1つ）、中間デモ（4つ）を作成でき、それぞれに最大160行のメッセージとBGMを設定できます。
*   **計算システム**: 命中判定やダメージ計算には、キャラクターのすばやさやアイテムの持ちやすさ、さらには「集中力（集中ポテンシャル）」が影響するリアルタイムな計算式が採用されています。

### 7. 演出・その他
*   **BGM**: 外部ツール『音楽ツクール』で作成したオリジナル曲（MML形式）を取り込み、ダンジョン内やデモで演奏させることができます。
*   **グラフィック**: キャラクター、アイテム、モンスター、壁、ドア、さらにはマウスカーソルやアイコンに至るまで、エディターで自作が可能です。
*   **リアルタイム処理**: ゲーム内の時間は「サイクル」単位で進行し、1歩の移動や攻撃などに特定のサイクルを消費します。

## だんだんダンジョンのUIの仕様

『だんだんダンジョン』のUI（ユーザーインターフェース）は、主にプレイ中の「メイン画面」、メンテナンスを行う「データ画面」、そしてゲーム作成時に使用する「各エディター画面」で構成されています。

### 1. メイン画面（探索・戦闘時）
冒険の基本となる画面で、以下の5つのウインドーで構成されています。

*   **3Dウインドー**: 画面中央の大部分を占め、ダンジョン内の風景を冒険者の視点で表示します。
*   **ステータスウインドー**: パーティメンバーの状態を表示します。名前のほか、**HP（青いバー）**、**MP（赤いバー）**、**集中力（緑のバー）**が視覚的に確認できます。
*   **アクションアイコンウインドー**: キャラクターが行動するためのアイコンが並びます。「取る」「見る」「攻撃」「魔法」「防ぐ」「精神統一」などのアクションを選択できます。
*   **移動アイコンウインドー**: パーティを移動させるための矢印アイコンが表示されます。前後左右の移動、回転、上り下りに対応しており、テンキーでの操作も可能です。
*   **メッセージウインドー**: 主人公やモンスターの行動、取得したアイテムの重さ、ダメージ計算の結果などをテキストで表示します。

### 2. データ画面（メンテナンス・設定）
メイン画面で3Dウインドーを**右クリック**すると切り替わる画面です。アイテムの整理やステータス確認に使用します。

*   **装備・持ち物**: 人型のアイコンに武器や防具をドラッグして装備させます。また、3個まで収納できる「ポーチ」と、最大24個収納できる「リュック」の管理ができます。
*   **魔法**: 習得した魔法のシンボルが並び、ここから魔法を唱えることも可能です。
*   **ステータス・プロフィール**: キャラクターの身長、体重、年齢といった詳細な能力値や経歴を確認できます。
*   **システム設定（オプション・ディスク）**: ゲームスピードの変更、アイテムの簡易表示設定、データのセーブなどを行います。
*   **ZZZ（睡眠）**: 安全な場所で休息を取り、傷ついた体や消費した魔力を回復させるコマンドです。

### 3. マップエディターの画面
ゲーム作成の要となるマップエディターには、作業を効率化するための独自のUIが用意されています。

*   **マップ表示**: プレイヤー視点の「3Dエディットモード」と、真上から見下ろした「2Dエディットモード」を切り替えて作業できます。
*   **設定メニュー**: 壁、ドア、鍵穴、スイッチ、モンスター、アイテムなど、フロアに配置したいパーツをリストから選択します。
*   **座標表示**: 現在編集しているフロア内での正確な座標（位置）と、向いている方向を表示します。
*   **作業アイコン**: 「取る」「見る」といった基本アイコンのほか、宝箱の中身確認や2D/3Dモードの切り替えアイコンが配置されています。

### 4. 各エディターの共通構成
キャラクターやアイテム、モンスターなどの作成画面では、画面が大きく2つに分かれています。

*   **一覧表**: 画面の左側に、登録されているデータ名がリスト形式で表示されます。
*   **作成画面**: 一覧から選択したデータの詳細なパラメーター（攻撃力、重さ、グラフィックなど）を設定するエリアです。

操作は基本的にマウスで行うよう設計されていますが、数値入力や移動など、キーボード（テンキーやファンクションキー）を併用することでより素早く操作できるようになっています。

## DeepGrid Studioの仕様

だんだんダンジョンの設計は引き継ぎつつも、当時のPC98の制約は取り払う。利用できるリソースや色数などの上限は取り払う。

### 上限値の扱い

上記「だんだんダンジョン概略仕様」に登場する数量上限は、あくまで**初期値(デフォルト)**であり、プロジェクト設定で拡張できるようにする。ハードコードせず、設定値として保持する。

| 項目 | 初期値(オリジナル準拠) |
| --- | --- |
| レベル数 | 14 |
| 1レベルあたりのフロア数 | 5 |
| フロアサイズ | 40×40ブロック |
| ドア種類数(1レベルあたり) | 2 |
| キャラクター登録数 | 20 |
| パーティ人数 | 4 |
| アイテム種類数 | 199 |
| アイテム配置数(1レベルあたり) | 1023 |
| モンスター種類数 | 56 |
| モンスター種類数(1レベルあたり) | 4 |
| モンスター配置数(1レベルあたり) | 48 |
| 魔法種類数 | 80 |
| イベントフラグ数 | 64 |
| イベント実行遅延 | 0〜63サイクル |
| デモのメッセージ行数 | 160 |

実装方針:

- これらの値はデータ構造上の固定長にせず、可変長コンテナ + 設定値による上限チェックで扱う。
- 設定はプロジェクト(作成するゲーム)ごとに保存し、エディター上から変更できるようにする。
- 上限を引き上げても既存データが壊れないこと(後方互換)を前提とする。上限の引き下げは、既存データが超過する場合は警告する。
- ID や個数を格納するフィールドは、オリジナルのビット幅(8bit 等)に縛られない十分な幅の型を使う。

### UIの方針

オリジナルのUI仕様(上記「だんだんダンジョンのUIの仕様」)は画面構成の参考とするが、**忠実な再現よりも使いやすさを優先**し、現代的で優れたデザインがあればそちらを採用する。

- **踏襲するもの(ゲームの構造として意味がある部分)**:
  - **グリッド単位の移動と90度単位の視点回転**(ダンジョンマスター風の一人称固定視点)。フリームーブやマウスルックにはしない
  - メイン画面の基本構成(3Dビュー + パーティステータス + アクション + 移動 + メッセージ)
  - エディターの「左に一覧・右に詳細」レイアウト
  - HP/MP/集中力のバー表示による直感的な状態把握
- **現代的に改める例**:
  - 解像度・ウインドウサイズは可変とし、レイアウトは追従させる(固定解像度前提にしない)
  - 「3Dウインドー右クリックで画面切替」のような発見しづらい操作は、明示的なボタンやタブ、ショートカットキーに置き換える
  - 移動操作はグリッド移動を前提に、WASDで前後移動・左右平行移動(ストレイフ)、Q/Eで90度回転という現代のグリッド式ダンジョンクローラー(Legend of Grimrock 等)標準のキー割り当てを第一とし、画面上の矢印アイコンは補助とする
  - 移動・回転の**見た目は滑らかに補間**する(1歩・90度回転を0.2〜0.3秒程度のイージング付きアニメーションで表現)。判定・位置の論理はあくまでマス単位のまま。キー押しっぱなしで次の1歩が途切れず連続するよう入力をバッファリングする。落下も同様に滑らかなアニメーションで表現する
  - ドラッグ&ドロップによるアイテム・装備の操作は踏襲しつつ、右クリックメニューやツールチップ(パラメーター表示)で補強する
  - エディターにはUndo/Redo、コピー&ペースト、検索・フィルタ、複数選択など現代のツールで当然の操作を備える
  - キーバインドは設定で変更可能にする
- 迷った場合は「オリジナルがどうだったか」ではなく「今のユーザーにとって使いやすいか」で判断する。


## 開発環境、ビルド方法、実行方法

Rust の Bevy を利用する。ビルド・実行環境は `mycity-simulator` プロジェクトの構成を引き継ぐ。

### 環境の前提

- ホスト: WSL2 (Ubuntu) + WSLg。GUI は Windows デスクトップに表示される。
- ビルドと実行は **docker 経路が標準**(ソフトウェア Vulkan/lavapipe)。
  docker イメージ `gaia-maker-build`(Rust 1.x bookworm + Bevy ビルド依存 +
  X11/Wayland/Mesa/Vulkan ランタイム)を mycity-simulator と共用する。
  イメージが無い場合はビルドスクリプトが `docker/Dockerfile` から自動ビルドする。
- cargo レジストリキャッシュは docker ボリューム `gaia-cargo-registry` を共用し、
  依存のダウンロード・ビルドを再利用する。

### 依存クレート

mycity-simulator で docker 環境での動作実績があるバージョンに合わせる。

- `bevy = "0.15"`(default-features 無効。`bevy_pbr`, `bevy_render`, `bevy_winit`,
  `bevy_asset`, `bevy_scene`, `bevy_gltf`, `png`, `x11`, `multi_threaded` など
  必要な feature のみ有効化)
- `bevy_egui = "0.33"`(エディター UI 用)
- `serde` + `ron`(ダンジョン・キャラ・アイテム等の定義データのシリアライズ)
- Rust edition 2024(Rust >= 1.85)

### ビルド方法

```sh
./docker/deepgrid-build.sh          # release ビルド(標準)
./docker/deepgrid-build.sh debug    # debug ビルド
```

- リポジトリを `/app` に bind mount して `cargo build` を実行する。
  `target/` はホスト側に生成される(所有者は root になる点に注意)。
- ホストの cargo(`~/.cargo/bin/cargo`)での **コンパイルチェックは可**
  (`cargo check` / `cargo clippy`)。ただし `target/` が root 所有のため、
  `CARGO_TARGET_DIR=<書き込み可能な別ディレクトリ>` を指定すること。
- ホストでの `cargo run` は **不可**(`libxkbcommon-x11-0` が無く起動時に panic)。
  実行は必ず docker または Windows ネイティブ経路で行う。

### 実行方法

```sh
./docker/deepgrid-run.sh            # release バイナリを WSLg 経由で実行
./docker/deepgrid-run.sh debug      # debug バイナリを実行
```

- WSLg のソケット(`/tmp/.X11-unix`, `/mnt/wslg`)をコンテナに渡し、
  Windows デスクトップにウィンドウを表示する。音声は PulseServer 経由。
- `WGPU_BACKEND=vulkan`(lavapipe)が既定。

### 検証用スクリーンショット

開発時の自動検証用に、環境変数でヘッドレス相当の確認を行えるようにする。

```sh
DEEPGRID_DEBUG_SHOT=1 ./docker/deepgrid-run.sh
```

- 起動後に所定のシーンを描画して `debug-shot.png` を出力し自動終了する。
- 生成物の mtime が今回の実行時刻であることを確認してから使うこと
  (過去の実行の残骸を新しい検証結果と取り違えない)。
- 新しい `DEEPGRID_*` 環境変数を追加したら、run スクリプトの `docker run` に
  `-e` での転送を追加すること。

### Windows ネイティブ GPU 実行

docker 経路は lavapipe(CPU レンダリング)のため、GPU(RTX 4070)で快適に
動かす・性能確認する場合は Windows ネイティブ .exe を WSL 内でクロスビルドし、
interop で起動する。

- 使い分け: **docker 経路 = 検証の基準(リファレンス環境)**、
  **Windows 経路 = GPU で快適に遊ぶ/性能確認**。
- 一回だけの導入(mingw は sudo 必要):
  ```sh
  ~/.cargo/bin/rustup target add x86_64-pc-windows-gnu
  sudo apt install gcc-mingw-w64-x86-64
  ```
  リンカ指定は `.cargo/config.toml` にコミットする。
- ビルド + 実行: `./scripts/deepgrid-run-win.sh`
  - `cargo build --release --target x86_64-pc-windows-gnu` を
    `~/.cache/deepgrid-target-win/` に出力し、exe をリポジトリ直下に
    コピーして起動する(exe は .gitignore する)。
  - `DEEPGRID_*` / `WGPU_*` 環境変数は WSL→Windows 境界を越えないため、
    スクリプトが `WSLENV` に列挙して橋渡しする(env を追加したら列挙にも追加)。
  - 起動ログの `AdapterInfo` が NVIDIA + Vulkan であることを確認する
    (lavapipe や Gl でないこと)。
